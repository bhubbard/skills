# Kumo UI Component Registry

The Kumo registry is the **source of truth** for all component APIs. It is auto-generated from TypeScript types and component variant definitions — not hand-written — so it is always accurate.

---

## What the Registry Is

- **URL:** [`https://kumo-ui.com/registry/`](https://kumo-ui.com/registry/)
- **Repo path:** `packages/kumo/ai/component-registry.json` (+ `.md` and `schemas.ts`)
- **Generated by:** `scripts/component-registry/` codegen (930+ line orchestrator, 13 sub-modules)
- **Purpose:** AI-readable component metadata for LLMs and code-generation tools

The registry is auto-generated via `pnpm codegen:registry` and committed to the `ai/` directory. The `ai/` directory is entirely machine-generated — never edit it directly.

---

## Registry Structure

Each component entry in the registry contains:

```json
{
  "name": "Button",
  "import": "import { Button } from \"@cloudflare/kumo\"",
  "description": "...",
  "props": {
    "variant": {
      "type": "\"primary\" | \"secondary\" | \"danger\" | \"ghost\"",
      "required": false,
      "default": "\"primary\""
    },
    "onClick": {
      "type": "(event: React.MouseEvent) => void",
      "required": false
    },
    "disabled": {
      "type": "boolean",
      "required": false,
      "default": false
    }
  },
  "variants": "KUMO_BUTTON_VARIANTS"
}
```

---

## Variant Constants (`KUMO_{NAME}_VARIANTS`)

Each component exports a `KUMO_{NAME}_VARIANTS` const with all valid variant values. These are **machine-readable and lint-enforced** — the registry references them as the authoritative list.

```typescript
// In the component source file:
export const KUMO_BUTTON_VARIANTS = {
  variant: ["primary", "secondary", "danger", "ghost"],
  size: ["sm", "md", "lg"],
} as const;

// Usage — always prefer const over hardcoded strings:
import { KUMO_BUTTON_VARIANTS } from "@cloudflare/kumo";
type ButtonVariant = typeof KUMO_BUTTON_VARIANTS.variant[number];
```

## How to Use the Registry

### 1. Browse Online
Visit [`https://kumo-ui.com/registry/`](https://kumo-ui.com/registry/) for human-readable component metadata with props, descriptions, and variant tables.

### 2. Reference `ai/component-registry.json` Directly
When you need the authoritative prop types for code generation or validation, reference the JSON directly:

```bash
# In a Kumo monorepo clone:
cat packages/kumo/ai/component-registry.json | jq '.components.Button'
```

### 3. Use `kumo doc` CLI
Get component docs directly in your terminal:

```bash
npx @cloudflare/kumo doc Button
npx @cloudflare/kumo doc Select
npx @cloudflare/kumo doc TimeseriesChart
```

### 4. Use `kumo ls` CLI
List all available components and blocks:

```bash
npx @cloudflare/kumo ls
```

## Using the Registry for AI Code Generation

The registry is explicitly designed for LLMs. When generating code with Kumo components:

1. **Always check variant values from the registry** — don't guess prop values
2. **Use `KUMO_{NAME}_VARIANTS` for type safety** — avoids invalid string literals
3. **Prefer the `import` field** — it gives the exact import statement to use
4. **Trust `default` values** — listed in the registry for all optional props

### Pattern for Reliable Component Code

```tsx
// ✅ Type-safe with registry variants
import { Button, KUMO_BUTTON_VARIANTS } from "@cloudflare/kumo";

type ButtonVariant = typeof KUMO_BUTTON_VARIANTS.variant[number];

function MyButton({ variant = "primary" }: { variant?: ButtonVariant }) {
  return <Button variant={variant}>Click me</Button>;
}

// ❌ Fragile — hardcoded string not validated at compile time
function MyButton() {
  return <Button variant="primay">Click me</Button>; // typo, no error
}
```

## Blocks vs Components in the Registry

The registry distinguishes:

- **Components** — exported from `@cloudflare/kumo`, in the registry, importable directly
- **Blocks** — NOT in the registry, NOT exported, scaffolded via `kumo add <BlockName>`

Blocks are complex patterns (PageHeader, ResourceList, DeleteResource) that get copied into your source code and customized. The registry only covers importable Components.

## Keeping the Registry Up to Date

If working in the Kumo monorepo:

```bash
# Regenerate registry after modifying component types or variants
pnpm codegen:registry

# Regenerate all auto-generated files (registry + themes + primitives)
pnpm codegen
```

Never hand-edit files in `ai/` — they will be overwritten on next codegen run.
